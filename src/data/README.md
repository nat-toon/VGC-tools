# `src/data/`

Generated data files consumed by the app.  Do not edit by hand - re-run
`npm run build:data` to regenerate.

## Top-level files

| File           | Format         | Source                                                  | Size  |
| -------------- | -------------- | ------------------------------------------------------- | ----- |
| `items.js`     | `Object.freeze` | `scripts/.cache/upstream/data/items.ts` + per-mod patches (fetched from GitHub) | 45 KB |
| `abilities.js` | `Object.freeze` | `scripts/.cache/upstream/data/abilities.ts` + per-mod patches (fetched from GitHub) | 20 KB |

The upstream TypeScript files are pulled by `scripts/fetch-showdown.cjs`
from `https://raw.githubusercontent.com/smogon/pokemon-showdown` and
cached in `scripts/.cache/upstream/`.  Set `POKEMON_SHOWDOWN_LOCAL` to
override with a local checkout.

`items.js` and `abilities.js` export a single `ENTRIES` object literal
keyed by id; the lib helpers convert these to `Map`s at import time so
lookups are O(1).

## `regulations/`

Per-regulation bundles, one file per regulation.  Generated by
`scripts/build-regulations.cjs`.

| File                  | Exports                                                                       | Size  |
| --------------------- | ----------------------------------------------------------------------------- | ----- |
| `regulations/{key}.js` | `KEY`, `LABEL`, `POKEMON` (Set), `ITEMS` (Set), `ABILITIES` (Set), `IS_NONSTANDARD` (Set), `LEARNSETS_URL` | ~5 KB |
| `regulations/index.js` | barrel re-exporting every bundle as a namespace                                | <1 KB |

A bundle is a self-contained, pure-data module - no functions, just Sets
and constants.  `lib/regulations.js` imports the barrel and assembles
the runtime `REGULATIONS` registry by attaching per-regulation helpers
(`isNonstandardMove`, `isNonstandardItem`, `isNonstandardAbility`) on
top of each bundle's `IS_NONSTANDARD` set.

`LEARNSETS_URL` points at the per-regulation learnset JSON in
`public/regulations/{key}/learnsets.json` (see "Per-regulation
learnsets" below).  The "all" pseudo-regulation has no bundle; its
`learnsetsUrl` is `null` and the lib falls back to the master
`public/learnsets.json`.

### ITEMS and ABILITIES are per-regulation

`ITEMS` and `ABILITIES` are per-regulation allowlists, NOT derived
from a globally-merged table.  Each regulation builds its own from:

```
scripts/.cache/upstream/data/{items,abilities}.ts  +  this regulation's modDir/{items,abilities}.ts
```

(Or from `POKEMON_SHOWDOWN_LOCAL/data/...` if set.)

Patches are applied on top of the master.  Items/abilities whose
final `isNonstandard` is `undefined` or `null` are legal.  This
prevents a real bug: an item un-banned in one regulation is NOT
automatically legal in another.

The global `items.js` and `abilities.js` (emitted by
`parse-ts-data.cjs`) are still used for name/desc lookups, but their
`isNonstandard` field is no longer authoritative for legality.

### Adding a new regulation

1. Append a `REGULATIONS` entry in
   `scripts/regulations-config.cjs`:

   ```js
   {
     key: 'gen-9-ou',
     label: 'Gen 9 OU',
     modDir: 'gen9',           // Pokemon Showdown mod folder
     isNonstandard: ['Past', 'LGPE', 'Future', 'CAP', 'Unobtainable'],
     excludedRosterNonstandard: ['Past', 'Future', 'LGPE', 'Unobtainable'],
   }
   ```

   See the top of `regulations-config.cjs` for the field reference and
   the meaning of `isNonstandard` / `excludedRosterNonstandard`.

2. Re-run `npm run build:data`.

The build pipeline will:
- Derive the roster from `mods/{modDir}/formats-data.ts`, excluding any
  species whose `isNonstandard` is in `excludedRosterNonstandard`
- Derive the per-regulation ITEMS allowlist from master `items.ts` +
  `mods/{modDir}/items.ts`
- Derive the per-regulation ABILITIES allowlist from master
  `abilities.ts` + `mods/{modDir}/abilities.ts`
- Derive the per-regulation learnset from
  `mods/{modDir}/learnsets.ts`, merged with form fallback to base
  species (so `Aegislash-Blade` uses `Aegislash`'s learnset, and
  `Rotom-Heat` gets `Rotom`'s moves PLUS its form-specific `Overheat`)
- Emit a new bundle at `regulations/{key}.js` and update
  `regulations/index.js` to re-export it
- Make the new entry the `DEFAULT_REG` in `lib/regulations.js` if it's
  the first one in the config

The new regulation becomes a dropdown option automatically - no UI
changes needed.

### Preserving old regulations across Showdown updates

The config is a list of frozen snapshots.  When Showdown updates to a
new regulation:

- **To add it without losing the old one**: append a new entry to the
  config with the new `key` and `modDir`.  The old entry stays in the
  list, so its bundle is re-emitted under the same `key` (or you can
  remove it - the file is still on disk and won't be re-emitted).  The
  `key` is the public id used in localStorage and the URL, so
  changing it would orphan any saved state.

- **To replace the old one**: just edit the existing entry's `key` and
  `modDir`.  A fresh bundle is emitted under the new `key`; the old
  file is left on disk but not in the barrel, so it won't be
  selectable.  Any localStorage references to the old `key` will fall
  back to the default.

### Frozen regulations

Set `frozen: true` on a regulation entry to mark it as "do not
rebuild from upstream".  The build will:

- skip reading from `mods/{modDir}/` (so `modDir` becomes optional)
- skip regenerating `src/data/regulations/{key}.js`
- skip regenerating `public/regulations/{key}/learnsets.json`
- still include the regulation in the bundle barrel
  (`src/data/regulations/index.js`), so it stays in the UI dropdown
- fail loudly if the bundle or learnset files don't already exist
  on disk (the entry was added but never built once)

Typical workflow for snapshotting a past season:

1. Add the entry as non-frozen.  Run `npm run build:data` to
   generate the bundle + learnset from upstream.
2. Edit the entry to add `frozen: true`.  Re-run `npm run build:data`.
   The existing files are preserved; future builds leave them alone.

To unfreeze: set `frozen: false` (or remove the field) and rebuild -
the bundle + learnset will be regenerated from upstream.
